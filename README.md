# **Custom Thread Pool**

Учебный проект по дисциплине **«Многопоточное и асинхронное программирование на Java»**.

Проект представляет собой собственную реализацию пула потоков с настраиваемыми параметрами, несколькими очередями задач, алгоритмом распределения **Round Robin**, обработкой отказов при перегрузке и подробным логированием работы.

---

## **Описание проекта**

В рамках работы был реализован собственный пул потоков без использования стандартного `ThreadPoolExecutor` в основной логике пула.

Реализованный пул позволяет:

- задавать базовое количество потоков;
- ограничивать максимальное количество потоков;
- задавать время простоя дополнительного потока;
- использовать ограниченные очереди задач;
- поддерживать минимальное количество резервных потоков;
- выполнять задачи через `execute(Runnable)`;
- выполнять задачи через `submit(Callable<T>)`;
- распределять задачи между очередями по алгоритму **Round Robin**;
- отклонять задачи при перегрузке;
- выполнять мягкое завершение через `shutdown()`;
- выполнять принудительное завершение через `shutdownNow()`;
- логировать основные события работы пула.

Также в проект добавлена демонстрационная программа и простой benchmark для сравнения с `ThreadPoolExecutor`.

## **Основные компоненты**

### **CustomExecutor**

Интерфейс управления пулом потоков.

```java
public interface CustomExecutor extends Executor {

    @Override
    void execute(Runnable command);

    <T> Future<T> submit(Callable<T> callable);

    void shutdown();

    void shutdownNow();
}
```

Он наследуется от стандартного `Executor`, поэтому пул можно использовать как обычный executor.

---

### **CustomThreadPool**

Основной класс проекта.

Он отвечает за:

- создание базовых потоков;
- создание дополнительных потоков при росте нагрузки;
- хранение списка рабочих потоков;
- распределение задач между очередями;
- обработку перегрузки;
- завершение простаивающих потоков;
- работу методов `execute`, `submit`, `shutdown`, `shutdownNow`.

При создании пула сразу запускается количество потоков, равное `corePoolSize`.

---

### **CustomThreadFactory**

Фабрика потоков создаёт потоки с уникальными именами:

```text
MyPool-worker-1
MyPool-worker-2
MyPool-worker-3
```

Также она логирует создание и завершение потоков.

Пример логов:

```text
[ThreadFactory] Creating new thread: MyPool-worker-1
[ThreadFactory] Thread finished: MyPool-worker-1
```

---

### **RejectionPolicy**

Интерфейс политики отказа.

Он определяет, что делать с задачей, если пул перегружен и больше не может принять новую задачу.

В демонстрационной программе используется стратегия отклонения задачи с выводом сообщения:

```text
[Rejected] Task overload-task-25 was rejected due to overload!
```

---

## **Алгоритм распределения задач**

В проекте используется алгоритм **Round Robin**.

Каждый рабочий поток имеет собственную очередь задач. При поступлении новой задачи пул выбирает следующую очередь по кругу.

Пример распределения:

```text
task-1 -> queue #1
task-2 -> queue #2
task-3 -> queue #3
task-4 -> queue #1
task-5 -> queue #2
```

---

## **Работа minSpareThreads**

Параметр `minSpareThreads` задаёт минимальное количество свободных потоков, которые пул старается поддерживать.

После добавления задачи пул проверяет, сколько worker-потоков сейчас свободны. Поток считается свободным, если он не выполняет задачу и его очередь пуста.

Если свободных потоков меньше, чем `minSpareThreads`, и общее количество потоков меньше `maxPoolSize`, пул создаёт дополнительный поток.

Например, при настройках:

```properties
corePoolSize=2
maxPoolSize=4
minSpareThreads=1
```

пул стартует с двумя базовыми потоками. Когда оба потока заняты выполнением задач, свободных потоков не остаётся. Пул создаёт дополнительный worker, чтобы сохранить хотя бы один резервный поток для новых задач.

---

## **Жизненный цикл worker-потока**

Каждый worker обрабатывает задачи из собственной очереди.

Если задача отсутствует, поток ожидает её в течение `keepAliveTime`.

Если за это время задача не поступила и общее количество потоков больше `corePoolSize`, worker завершается.

Пример логов:

```text
[Worker] MyPool-worker-4 idle timeout, stopping.
[Worker] MyPool-worker-4 terminated.
```

Базовые потоки в количестве `corePoolSize` не завершаются по таймауту простоя до вызова `shutdown`.

---

## **Завершение работы пула**

В проекте реализованы два способа завершения.

### **shutdown()**

Мягкое завершение.

После вызова `shutdown()` пул больше не принимает новые задачи, но продолжает выполнять задачи, которые уже были добавлены в очереди.

### **shutdownNow()**

Принудительное завершение.

После вызова `shutdownNow()` пул переводится в состояние завершения, рабочие потоки прерываются, а очереди задач очищаются.

## **Демонстрация работы**

В классе `Main` реализовано несколько сценариев.

### **Нормальная нагрузка**

В пул отправляется 8 задач:

```java
for (int i = 1; i <= 8; i++) {
    pool.execute(new DemoTask("normal-task-" + i, 1500));
}
```

Каждая задача имитирует работу через `Thread.sleep`.

Также демонстрируется выполнение задачи через `submit(Callable<T>)`:

```java
Future<String> future = pool.submit(() -> {
    Thread.sleep(1000);
    return "Result from Callable task";
});
```

Результат получается через:

```java
future.get();
```

---

### **Перегрузка пула**

Для демонстрации перегрузки отправляется 40 задач:

```java
for (int i = 1; i <= 40; i++) {
    pool.execute(new DemoTask("overload-task-" + i, 3000));
}
```

Так как количество задач превышает возможности пула и его очередей, часть задач отклоняется:

```text
[Rejected] Task overload-task-25 was rejected due to overload!
```

## **Анализ производительности**

Для сравнения производительности был добавлен benchmark-сценарий.

В нём одинаковое количество задач отправляется сначала в собственный `CustomThreadPool`, а затем в стандартный `ThreadPoolExecutor`.

Параметры теста:

| Параметр | Значение |
|---|---:|
| `corePoolSize` | 2 |
| `maxPoolSize` | 4 |
| `queueSize` | 5 |
| `keepAliveTime` | 5 секунд |
| `minSpareThreads` | 1 |
| Количество задач | 30 |
| Время выполнения одной задачи | 1500 мс |

Результат одного запуска:

| Пул потоков | Время выполнения | Отклонено задач |
|---|---:|---:|
| `CustomThreadPool` | 9046 мс | 6 |
| `ThreadPoolExecutor` | 9069 мс | 6 |

В данном запуске `CustomThreadPool` оказался быстрее на 23 мс. Однако разница небольшая, поэтому корректнее говорить, что оба пула показали сопоставимое время выполнения.

Основное время в тесте занимает искусственная задержка `Thread.sleep(1500)`, а не вычисления. Также нужно учитывать, что `CustomThreadPool` выводит подробные логи, что создаёт дополнительные накладные расходы.

Данный benchmark не является полноценным промышленным тестом производительности, но подходит для учебного сравнения поведения двух пулов при одинаковой нагрузке и одинаковых ограничениях.

Полный вывод демонстрационной программы и benchmark-сценария сохранён в файле:

```text

```

## **Мини-исследование параметров пула**

В ходе проверки были рассмотрены разные значения параметров пула.

| Параметр | Влияние на работу |
|---|---|
| `corePoolSize` | Чем больше значение, тем больше потоков создаётся сразу при запуске пула |
| `maxPoolSize` | Чем больше значение, тем больше задач может выполняться параллельно |
| `queueSize` | Чем больше значение, тем меньше отказов, но тем дольше задачи могут ожидать выполнения |
| `keepAliveTime` | Чем больше значение, тем дольше дополнительные потоки остаются активными после нагрузки |
| `minSpareThreads` | Чем больше значение, тем активнее пул создаёт резервные свободные потоки |

Для основной демонстрации были выбраны параметры:

```properties
corePoolSize=2
maxPoolSize=4
keepAliveTime=5
timeUnit=SECONDS
queueSize=5
minSpareThreads=1
poolName=MyPool
```

Такие значения позволяют показать все ключевые сценарии:

- старт базовых потоков;
- распределение задач между очередями;
- создание дополнительных потоков;
- работу резервных потоков;
- завершение простаивающих потоков;
- отказ задач при перегрузке;
- корректное завершение через `shutdown()`.

При увеличении `corePoolSize` пул сразу создаёт больше базовых потоков. Это позволяет большему количеству задач начать выполнение сразу, но увеличивает количество постоянно работающих потоков.

При увеличении `queueSize` уменьшается количество отказов, так как больше задач помещается в очереди. Однако задачи могут дольше ожидать выполнения.

При уменьшении `queueSize` пул быстрее начинает отклонять задачи. Это позволяет быстрее обнаружить перегрузку, но может приводить к большему количеству отказов.

При `minSpareThreads=0` дополнительные потоки создаются только при заполнении очередей. При `minSpareThreads=1` пул создаёт дополнительный поток раньше, когда свободных потоков становится меньше заданного значения.

Наиболее сбалансированным вариантом для демонстрации оказался набор параметров:

```properties
corePoolSize=2
maxPoolSize=4
queueSize=5
minSpareThreads=1
```

Он позволяет показать нормальную работу, перегрузку, создание дополнительных потоков и завершение лишних потоков по таймауту.
